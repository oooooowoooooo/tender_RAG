安徽/合肥招标方智能招采辅助系统，是一个面向招标人/采购人实际工作场景构建的高合规智能问答与检索辅助系统。项目围绕 政策规则解释、同类项目对标、企业主体核验 三类高频任务展开，基于全网公开信息爬取后的政策文件、招标公告与企业基本信息，构建了一套 以证据优先、受约束状态机驱动、可回归评测 的 Workflow-based RAG 系统。项目目标是为招标方在标前策划、招标文件编制、历史项目参考和主体信息核验等环节提供可追溯、可解释、可拒答、可迭代的智能辅助能力。

在数据处理层，项目对全网爬取后的异构文本进行统一清洗、去重、切分与结构化建模。首先基于 Python ETL 流水线完成文本抽取、编码清洗、字段归一、重复文档识别和脏数据过滤；随后构建统一 Schema，将政策类、公告类、企业类数据映射为统一的数据对象，并对政策文本额外补充地域标签、效力状态、生效时间、文种、条款编号 等关键元数据。

在知识组织与索引层，项目采用 “结构化存储 + 全文检索 + 向量索引” 的架构。结构化字段存储在 PostgreSQL 中，承担地区、公告类型、时间区间、主体名称、项目类别等精确过滤任务；全文索引基于 Elasticsearch / BM25 体系，负责法规标题、条款关键词、资格条件、评审办法、预算金额等强关键词命中场景；语义检索基于 Milvus + BGE 系列 Embedding 模型 构建向量索引，用于处理用户自然语言表达下的语义近似匹配。

在检索治理层，项目引入了一个轻量但非常关键的 Policy Priority Filter。该模块在多路召回之后，对候选政策证据按 地域匹配、效力状态、时间优先级、文种优先级、语义相关度 进行二次排序，优先保留同区域、现行有效、发布时间更近、适用范围更匹配的条款片段。

在编排与执行层，项目没有把系统包装成完全自主规划的通用 Agent，而是采用 受约束状态机驱动的 Workflow-based RAG。整个链路由 Intent Parse → Hybrid Retrieve → Evidence Verify → Evidence-first Generate → Self-check / Refuse 五个核心节点组成。首先，系统对用户问题做任务识别和槽位抽取，判断其更偏向政策问答、历史项目对标、企业主体查询，还是多任务组合问题；随后进入混合检索阶段，从结构化过滤、全文召回、向量召回三条链路并行获取候选证据，并通过 Reranker 做统一重排；接着由 Verify 节点检查证据是否足够、政策是否有效、地域是否匹配、知识边界是否越界；只有在证据通过校验后，才进入生成节点输出回答；最后由 Self-check 节点检测是否存在引用缺失、证据覆盖不足、结论越权等问题，必要时触发二次放宽检索或降级为保守回答/拒答。

在生成与回答层，系统坚持 Evidence-first Answering 原则。LLM 不是直接基于问题自由发挥，而是严格围绕命中的政策条款、招标公告片段和企业基础信息作答，输出内容默认包含结论、证据来源、适用条件与风险提示 四个组成部分。对于超出安徽/合肥知识边界、命中证据不足、区域不匹配或信息存在冲突的请求，系统不会强行生成，而是触发保守回答或拒答策略。

在重排层，系统采用 BGE Reranker 级别的交叉编码器模型 对多路召回结果进行精排，用于统一处理政策片段、公告段落和企业信息摘要之间的相关性比较。召回阶段先保留各链路 TopK，完成候选去重和基础归一化后，再交由 Reranker 输出最终排序结果。

在评测层，项目专门设计了一个 Automated Harness，将评测样本、召回参数、Prompt 版本、Reranker 版本、阈值配置和模型版本全部解耦，支持离线批量回放、回归测试和 badcase 持续沉淀。评测体系采用分层设计：第一层评估检索能力，包括 Recall@K、MRR、引用命中率、候选覆盖率；第二层评估规则与路由能力，包括地域过滤正确率、过期条款拦截率、Wrong Routing Rate、拒答边界命中率；第三层评估答案能力，结合 LLM-as-a-Judge + 人工抽检 对答案忠实性、引用精度、拒答合理性进行混合判断。

在服务化与工程层，项目采用 Python + FastAPI 对外提供统一问答接口和检索接口，支持结构化检索、问答生成、trace 查询与评测触发。推理层可对接本地部署的中文大模型，如 Qwen3 8B系列，推理引擎可采用 vLLM / Transformers；Embedding 与 Reranker 独立部署，以降低不同模块之间的耦合度。项目同时设计了最小生产化机制：通过 Redis 实现热点查询缓存与短 TTL 结果复用；当向量服务异常时自动降级为 BM25 + 结构化过滤；当生成模型超时或不可用时，退化为“证据列表 + 模板化摘要”输出，而不是整链路失败；全链路产出 trace_id、query、route、retrieved_chunk_ids、prompt_version、output、refusal_reason、latency_ms 等审计日志，便于后续回放、定位和 badcase 复盘。

项目形成完整闭环：
全网公开数据 → ETL 清洗与统一 Schema → 元数据增强与切块 → PostgreSQL / Elasticsearch / FAISS 多路索引 → 混合检索与优先级过滤 → Reranker 精排 → 状态机编排 → 证据优先生成 → 自检拒答 → Harness 回归评测 → API 服务化与 Trace 审计。
